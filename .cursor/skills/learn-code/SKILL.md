# 學寫法（Learn Code）

把「**剛討論過的一個程式寫法**」沉澱成一頁可重用技巧筆記：

- **一檔一技巧**，聚焦最核心那一個寫法
- **抽象化骨架**，去商業命名，換情境也能套
- **只談怎麼寫**，不解釋商業邏輯
- 用 **context7 / microsoft-learn 真比對正確性**，附官方網指

## 觸發

| 寫法                   | 行為                                                            |
| ---------------------- | --------------------------------------------------------------- |
| `/learn-code`          | 掃當前對話 → 列技巧候選＋標推薦 → 你挑一個 → 產檔               |
| `/learn-code <關鍵字>` | 直接鎖定你指的技巧（如 `/learn-code GroupBy 那段`），跳過列清單 |

## 輸出

| 項目   | 規則                                                    |
| ------ | ------------------------------------------------------- |
| 位置   | `docs/learn-code/<技巧英文名-kebab>.md`（不存在則建立） |
| 檔名   | 技巧英文名 kebab-case，例：`groupby-lookup-table.md`    |
| 同名檔 | **先停下問你**：蓋掉 / 改名 / 取消（不自動覆蓋）        |

## 執行流程

```
1. 掃當前對話 → 抽出「程式技巧候選」
2. 有關鍵字 → 直接鎖定；無 → 列候選＋標推薦 → 等你挑
3. 重新讀實檔，確認該寫法的精確語法
4. 抽象化成通用骨架（去商業命名）
5. 驗證分流核對正確性 → 附網指
6. 核對有誤 → 用正解改寫骨架 + 關鍵點標註
7. 套五段模板
7.5 砍一遍：逐行跑最小技巧測試（#2），不是技巧本身就刪
8. kebab 檔名 → 寫入 docs/learn-code/；同名先問
```

### Step 1：抽技巧候選

從當前對話找出**值得記的程式寫法**（語法特性、API 用法、idiom、資料結構運用）。

- **排除**：商業邏輯、瑣碎流程控制（單純 `if`/`for`/變數賦值）
- 一個寫法 = 一個候選

### Step 2：鎖定一個

- **有關鍵字** → 直接對應到那個候選
- **無關鍵字** → 列候選清單，**標出推薦那個**，等使用者挑：

  ```
  掃到這幾個寫法：
  1. ⭐(推薦) GroupBy + ToDictionary 建查找表
  2. record 不可變物件
  要學哪一個？
  ```

### Step 3：讀實檔

**重新讀相關程式碼檔**確認精確語法——**絕不憑對話記憶複製片段**（會錯/會過時）。對話沒明確指到檔就問。

### Step 4：抽象化

把實檔片段改寫成**通用骨架**：

- 商業命名 → 通用名（`cardHolderDict` → `lookup`、`IdNo` → `key`）
- 去掉專案專屬脈絡，只留寫法本體
- **不留出處連結**（全脫鉤，成為可分享的純技巧）

### Step 5：驗證分流（核對正確性）

判斷技巧屬性，走對應來源**比對寫法對不對**，不是只貼連結：

| 技巧屬性                                                 | 驗證來源                                            |
| -------------------------------------------------------- | --------------------------------------------------- |
| 第三方套件（FusionCache、Dapper…）                       | **context7**（`resolve-library-id` → `query-docs`） |
| C# 語言特性 / .NET BCL（LINQ、record、pattern matching） | **microsoft-learn**（`microsoft_docs_search`）      |
| 前兩者都查不到                                           | **web search** 收尾                                 |
| 仍查不到                                                 | 標 `⚠️ 未找到官方文件佐證`，不硬掰連結              |

### Step 6：發現錯誤就修正

核對後若骨架寫法有誤（過時 / 簽章錯 / 幻覺）：

- **「寫法」段直接用正解骨架**
- **「關鍵點」標一行**：`原 X 寫法已過時，改用 Y`

### Step 7：套模板輸出

見下方「輸出模板」。

### Step 7.5：砍一遍（寫檔前強制）

對「寫法」段**逐個函式、逐行**跑最小技巧測試（硬規則 #2）：拿掉它，這個技巧還示範得出來嗎？能就刪。重點抓兩種最常見的膨脹：

- **重複變體**——有沒有第二個方法/區塊在示範同一招（只差 GET/POST、單一/批次）？留一個，其餘併進關鍵點。
- **混入鷹架**——有沒有別篇技巧的東西混進來（建構子、註冊、前後置、wrapper、未用 import）？砍掉。

> 砍完才進 Step 8。骨架理想是「一個核心 + 最少示範」，不是「一個可運作的完整類別」。

### Step 8：寫檔

kebab 檔名 → `docs/learn-code/`；同名先問（Step 同名規則）。

## 硬規則

1. **一檔只一個技巧**——掃到多個先列候選讓使用者挑，不塞合輯
2. **最小技巧測試（核心判準）**——doc 只放「示範這一個技巧所需的最小 code」。每一行都自問：**這行是技巧本身嗎？不是就刪。** 技巧以外的一切（專案脈絡、周邊流程、重複變體…）都不進 doc。
3. **只談怎麼寫**——✅「GroupBy 建表查找 O(1)」 ❌「為什麼要對照持卡人」
4. **片段不憑記憶**——一定回讀實檔確認語法
5. **驗證是真核對**——不只查 API 存不存在、簽章對不對，更要查**用法對不對**（解析規則、邊界行為、易踩的陷阱…）；查到陷阱就收進關鍵點。不是只貼連結
6. **查不到就明說**——標 `⚠️ 未找到官方文件佐證`，不杜撰
7. **極簡**——五段內講完，沒內容的關鍵點不硬湊

## 輸出模板

```markdown
# [技巧名稱，如「GroupBy 建查找表取代巢狀迴圈」]

> 一句話：解決什麼**寫法**問題（不碰商業）。例：避免兩份清單比對時的 O(n²) 巢狀迴圈。

## 寫法

​`csharp
var lookup = source.GroupBy(x => x.Key)
                   .ToDictionary(g => g.Key, g => g.First());
if (lookup.TryGetValue(id, out var hit)) { ... }
​`

## 關鍵點

- GroupBy + ToDictionary 一次建表，查找 O(1)
- 重複 key 要先決定取哪筆（First / Last）
- 只讀不改用 ILookup 更貼語意

## 官方網指

- [Enumerable.ToDictionary — Microsoft Learn](https://learn.microsoft.com/dotnet/api/system.linq.enumerable.todictionary)
```

> 查不到官方頁面時，「官方網指」段改寫一行：`⚠️ 未找到官方文件佐證`。
