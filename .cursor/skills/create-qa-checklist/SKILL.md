# 從 logic flow 產出人工驗收 checklist（QA Checklist）

## 使用時機

使用者給你**一份 logic flow 文件**（通常在 `docs/features/<name>.md`，由 `map-feature-flow` 產出），希望產出對應的「人工驗收 checklist」。

例如：
- 「幫 `docs/features/login.md` 產驗收清單」
- 「把這個 feature flow 變成 QA todo」
- 「我要登入流程的人工測試清單」

## 輸入與輸出

| 項目     | 說明                                                                                  |
| -------- | ------------------------------------------------------------------------------------- |
| 輸入     | logic flow 文件路徑（一次一個），可選模式 `api` / `web` / `full`（預設）              |
| 輸出     | `docs/qa/<name>[.api\|.web].md`（不存在則建立資料夾）                                 |
| 檔名規則 | 沿用輸入 flow 檔名 kebab-case，依模式加後綴：full→無後綴 / api→`.api` / web→`.web`    |

## 三種模式

| 模式   | 範圍                                                                | 適用場景                          | 輸出檔名範例     |
| ------ | ------------------------------------------------------------------- | --------------------------------- | ---------------- |
| `api`  | 只列後端驗收（HTTP status、回傳 schema、後端副作用、DB 寫入 / Log） | Postman / API integration 人工測 | `login.api.md`   |
| `web`  | 只列前端驗收（UI 動作、表單檢查、toast、導頁、cookie 寫入）         | UI 人工驗收（後端可 mock）        | `login.web.md`   |
| `full` | 全鏈路 e2e（前端動作 + 後端 status + DB / cookie 證據鏈）           | 完整功能驗收（預設）              | `login.md`       |

> 使用者沒明說 → 預設 `full`；要確認可問「要 `api` / `web` / `full`？」

## 執行流程

### Step 0：抓 repo root

跑 `git rev-parse --show-toplevel` 取 repo 絕對路徑（如 `D:/程式/Lab.Todo`）。**所有連結 URL 用 `file:///{repo-root}/...#L{line}` 為前綴**（同 `map-feature-flow`，理由見硬規則 #6）。

### Step 1：讀 logic flow 文件

完整讀完輸入文件，**逐章節掃描**並標記：

| 來源章節             | 拆成 todo 的對象（章節歸屬見硬規則 #5）                                   |
| -------------------- | ------------------------------------------------------------------------- |
| 入口                 | 每種「怎麼被打開」（直接 URL → 正常流程；路由守衛 / 攔截器 → 進出頁面）   |
| 使用者操作           | 每個 `@click` / `@submit` 等 handler 觸發點                               |
| 呼叫的 API           | 每個 endpoint 的成功 / 失敗 status code                                   |
| 做了什麼判斷         | 每條「條件 → 動作」拆成獨立 todo（一條 = 一個 if 分支）                   |
| 成功後做什麼         | 每個 side-effect（cookie 寫入、導頁、toast、寫 DB）                       |
| 副作用               | 每個項目（Log、Email、Event、Cache 等）                                   |

### Step 2：判斷實作狀態（決定 `[x]` 還是 `[ ]`）

每個 todo 項目去找對應實作：

| 情況                                                       | 結果        | 註記                          |
| ---------------------------------------------------------- | ----------- | ----------------------------- |
| logic flow 已附行號連結，開檔確認程式碼存在且邏輯對得上    | `[x]`       | 直接用 flow 文件的連結         |
| logic flow 沒附連結（例如 UI 元素），grep 確認元素存在     | `[x]`       | 附 grep 找到的檔案位置         |
| logic flow 明確寫「目前無實作」「純 UI」「TODO」           | `[ ]`       | 註記「純 UI / 未實作」+ UI 位置 |
| logic flow 提到但程式碼 grep 不到                          | `[ ]`       | 註記「實作缺失 / 待確認」      |
| **不在 flow 文件但屬常見邊界**（如「快速雙擊」「超長輸入」）| `[ ]`       | 放「額外建議測」章節           |

> 不確定就明說：「我推測這條已實作，請手動 grep `Xxx` 確認」

### Step 3：按模式過濾

| 模式   | 砍掉的章節                                              |
| ------ | ------------------------------------------------------- |
| `api`  | 砍 UI 動作 / 表單前端檢查 / toast / 導頁 / cookie 章節 |
| `web`  | 砍後端 DB 寫入 / 後端 Log / JWT 簽發內部邏輯章節       |
| `full` | 不砍，全列                                              |

> 但「跨層」的項目（如 `401 → 自動導 /login`）兩邊都要：`api` 列「API 回 401」、`web` 列「收到 401 後導頁」

### Step 4：套用模板輸出

見下方「輸出模板」。

### Step 5：寫入檔案

寫到 `docs/qa/<name>[.api|.web].md`。`docs/qa/` 不存在則建立。

## 硬規則（最重要）

1. **全列、實作完才打勾**
   - logic flow 提到的每件事都列成 todo
   - 已實作（程式碼找得到對應邏輯）→ `[x]`
   - 未實作（純 UI / 找不到實作 / 純粹建議補測）→ `[ ]`
   - **不要砍「沒做的事」**：純 UI 也要列，只是不打勾

2. **一條 todo = 一個動作 + 一個可觀察的斷言**
   - ✅ `[ ] 帳號空白送出 → 欄位下方顯示「帳號為必填」訊息，且 Network 無 /user/login 請求`
   - ❌ `[ ] 測試表單檢查`（沒動作、沒斷言）
   - ❌ `[ ] 點 Sign In`（沒斷言）

3. **每條附「證據」與「實作位置」**
   - **證據**：去哪裡看（DevTools Network / Application Cookies / toast 顯示文字 / URL bar / DB 查詢 / Log 檔）
   - **實作位置**：
     - 已實作 `[x]` → 必附對應程式碼連結
     - 未實作 `[ ]` → 寫「未找到（理由）」**並掛 UI 元素本身的位置連結**（讓 QA 能直接點開看那個 checkbox / link 長在哪），例：`未找到（flow 標註「純 UI」）— [Login.vue:172](...)`
   - 格式：todo 主體一行 + `>` 引用 2 行（證據 / 實作）
   - **引用縮排用 4 個 space**（避免 Cursor / GitHub markdown 渲染走樣）

4. **入口段每種進入方式都對應一條驗收**
   - flow 文件「入口」段列的每種「怎麼被打開」（直接輸入 URL / 路由守衛導入 / 攔截器導入 / 從某頁跳轉）都要產出對應驗收 todo
   - 「直接輸入 URL」最容易被漏，但這是 QA 第一步——必須有一條像「直接訪問 `/login` → 顯示表單元素（帳號 / 密碼 / 按鈕）」

5. **章節按使用者旅程組織，章節名用口語（不用技術術語）**
   - 順序：`正常流程` → `表單檢查` → `錯誤情境` → `進出頁面` → `順帶會發生的事` → `沒做的功能` → `額外建議測`
   - 沒內容的章節整個砍掉，不留空殼
   - 「沒做的功能」獨立成章節，**不要混在主 flow 裡**（避免測試者卡在「為什麼點了沒反應」）
   - **不准用**「Golden Path」「副作用」「Side Effect」「邊界」「守衛」「Edge Case」這類工程術語當章節名（給 QA 看的，他們不需要懂這些詞）

6. **檔案連結用 markdown 連結（絕對路徑 + `#L` 行號），行號要精確**
   - 格式：`[檔名:行號](file:///{repo-root}/{相對路徑}#L{行號})`
   - 例：`[Login.vue:25](file:///D:/程式/Lab.Todo/src/todo-web/src/views/Login.vue#L25)`
   - `{repo-root}` 由 Step 0 取得
   - **行號精確到對應邏輯所在那一行**，不是段落起始：
     - ✅ Zod schema 內 `userId: z.string().trim().min(1)` 在 25 行 → 寫 `:25`
     - ❌ Zod resolver 起始 `zodResolver(z.object({...}))` 在 23 行 → 不要寫 `:23`（那是包裝層，斷言對應的程式碼在內部那行）
     - ✅ `BusinessLogicError` 那一行 → 指到那行；不要指到整個 handler 起始
   - **理由**：Cursor markdown preview 只認 `file://` + `#L行號`，且 QA 點進去要直接看到對應邏輯，不要再往下捲

7. **斷言要可觀察、能精確就精確、可從程式碼讀到的字串要寫進斷言**
   - ✅「Toast summary『登入失敗』、detail『帳號或密碼不正確，或者登入已過期』」（精確到實際顯示文字）
   - ✅「Network 看到 `POST /user/login` 回 `422`」「Cookies 有 `token`，`Max-Age` ≈ `expiresIn × 60`」
   - ❌「應該會跳錯誤」「登入要成功」（無法判定 pass/fail）
   - ❌「跳 toast」（沒寫 toast 文字，QA 還要回去問開發）
   - 條件 / 預期值能引用 flow 文件原句就引用（如 `code === 2000`、toast 顯示文字、API 回傳 message）
   - **凡是程式碼裡能讀到的字串（toast、error message、status code 數字）一律寫進斷言**，不要讓 QA 二次查證
   - **toast 斷言要寫全部可見段落**：PrimeVue / Element Plus / AntD 等的 toast 通常有 `summary + detail`（或 `title + message`）兩段，**有幾段就寫幾段**，不要只寫 summary
     - ✅「toast summary『網路連線異常』、detail『無法連線至伺服器，請確認網路狀態或稍後再試』」
     - ❌「toast『網路連線異常』」（漏 detail，QA 不知道要驗第二段）

8. **不要重複 logic flow 的解釋**
   - flow 文件已寫「`code === 2000` 才寫 cookie」→ todo 寫「成功登入後 Cookies 有 `token`」即可，不重述為什麼
   - flow 文件已連程式碼 → todo 沿用連結，不另起爐灶

9. **「額外建議測」獨立章節**
   - logic flow 沒明確要求但實務上應驗證（如 race condition、極端輸入、權限切換、瀏覽器相容）→ `## 額外建議測`
   - 每條一樣 `[ ]` + 證據 + **理由**（為什麼建議測）

10. **模式不同 → 章節不同，但同條 todo 寫法一致**
    - 不要為 `api` 模式重寫 todo；該條 todo 在 `web` 模式長什麼樣，在 `api` 模式就砍掉或留 API 視角的同條

11. **不確定就明說**
    - 「此項實作未在 flow 中標明位置，我推測在 X，請確認」

## 輸出模板

> 模板**框架無關**，範例為示意。**章節按需出現**：沒內容的章節直接砍。
>
> **連結格式**：下方模板 `(路徑)` 為簡寫示意；實際輸出時所有連結 URL 必須是 `file:///{repo-root}/{相對路徑}#L{行號}`（見硬規則 #6 與 examples.md）。
>
> **引用縮排**：todo 下方 `>` 引用統一用 4 個 space 起頭（見硬規則 #3）。

````markdown
# [功能中文名稱] - 驗收清單

> **模式**：全鏈路（full） / 後端（api） / 前端（web）
> **來源 flow**：[docs/features/<name>.md](路徑)
> **產出時間**：YYYY-MM-DD
> **圖例**：`[x]` 程式碼已實作（請手動驗證行為正確）；`[ ]` 未實作 / 純 UI / 待補測

## 正常流程

- [x] 直接訪問 `/path` URL → 顯示頁面主要元素（列舉具體欄位 / 按鈕）
    > **證據**：肉眼觀察頁面元素
    > **實作**：[檔案:行號](路徑)
- [x] [動作] → [可觀察的預期結果，含具體文字 / status code]
    > **證據**：DevTools Network / Cookies / toast / URL / DB 查詢
    > **實作**：[檔案:行號](路徑)

## 表單檢查

- [x] [輸入條件] → [預期錯誤訊息 / 阻擋行為]
    > **證據**：欄位下方訊息文字、Network 是否發出請求
    > **實作**：[檔案:行號](路徑)

## 錯誤情境

- [x] [錯誤觸發條件] → [預期 toast 文字 / status code / 停留頁面]
    > **證據**：[要看的地方]
    > **實作**：[檔案:行號](路徑)

## 進出頁面

- [x] [守衛 / 攔截觸發條件] → [預期導頁 + toast 文字]
    > **證據**：URL bar、toast、`route.query.redirect` 值
    > **實作**：[檔案:行號](路徑)

## 順帶會發生的事

- [x] [副作用名稱] → [可驗證的存在證據]
    > **證據**：Log 檔位置 / DB 查詢 / Email 信箱 / Event log
    > **實作**：[檔案:行號](路徑)

## 沒做的功能

- [ ] [UI 元素] → [若實作預期會怎樣]
    > **證據**：N/A（目前無對應實作）
    > **實作**：未找到（flow 文件標註「純 UI」）— [UI 元素位置](路徑)

## 額外建議測

> flow 文件未明確要求，但實務上建議手動驗證。

- [ ] [邊界 / race / 相容性測試]
    > **證據**：[預期觀察點]
    > **理由**：[為什麼建議補測]
````

## 注意事項

- **跨層的條件分支**（如 `401 → 自動導 /login`）`api` 模式列「API 回 401」、`web` 模式列「收到 401 後 cookie 清空 + 導頁」、`full` 兩者都列
- **同一 endpoint 多種失敗（400 / 401 / 403 / 422 / 500）** 一條一個，不要合併
- **DB 寫入 / Log** 必須給可驗證的查詢或檔案位置（不要寫「應該會寫入」）
- **完整範例見** [examples.md](examples.md)

## 自我擴充原則

每次跑完 skill 產出 checklist 後，**主動檢查**是否有「目前 skill 規則未明確涵蓋」的判斷或處理。若有，**立即執行以下三步**：

1. **擴充 skill 規則**
   - 能歸納進現有條目 → 加成子項目或範例庫
   - 完全是新類型 → 開新條目
   - 寫法：判斷原則 + 掃描步驟 + 範例，避免「為單一案例新增整條規則」

2. **同步 `.cursor/skills/create-qa-checklist/`**
   - SKILL.md 與 examples.md 兩邊內容必須一致

3. **記錄到 `.claude/rules/decision-log.md`**
   - 新增一筆，含：對照來源 / 新增規則 / 影響
   - 日期用當天

### 何時觸發

- **對照標準產出時**：使用者既有的 checklist 跟我產出的有差異 → 校準
- **產出全新 checklist 時**：自己審視「有沒有寫進 checklist 但 skill 沒明確要求」的判斷 → 把這個判斷抽象成規則
- **連續 3 個產出無新規則** → 停止主動擴充，避免無限膨脹
